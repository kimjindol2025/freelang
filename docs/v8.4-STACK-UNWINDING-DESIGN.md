# FreeLang v8.4: Stack Unwinding & Frame Cleanup

## 📌 개요

v8.3에서 "지금 당장 튈 것인가"(PC Redirect)를 구현했다면, v8.4는 **"튈 때 무엇을 버릴 것인가"**(스택 정리)를 구현하는 단계입니다.

**핵심 원칙**: 탈영(Escape)과 철수(Retreat)의 차이
- 탈영: 주소만 옮김 (위험, 메모리 누수)
- 철수: 스택을 정리하며 옮김 (책임감, 메모리 일관성)

---

## 🏗️ 아키텍처 설계

### 1. 스택 구조 (Stack Layout)

```
현재 상태 (THROW 지점):
┌─────────────────┐ ← SP_current (현재)
│  Depth3 locals  │   (777)
├─────────────────┤
│  Depth2 locals  │   (888)
├─────────────────┤
│  Depth1 locals  │   (666)
├─────────────────┤
│  TRY frame      │
├─────────────────┤
│  Main frame     │
└─────────────────┘ ← SP_saved (v8.2에서 저장)

THROW 발생!
→ UNWIND: Depth3, 2, 1 프레임을 역순으로 파괴
→ 최종: SP == SP_saved (원래 높이로 복구)
```

### 2. 언와인딩 메커니즘 (Unwinding Mechanism)

```
Phase 1: Iterative Popping
  현재 FP → 이전 FP → 이전 FP ... → TRY 프레임의 FP까지

Phase 2: SP Realignment
  SP_current를 SP_saved로 강제 변경
  = 중간에 쌓인 모든 스택 프레임 소거

Phase 3: Resource Safe-check
  각 프레임의 지역변수(원시값)를 메모리에서 제거
  (향후 v9.0에서 객체/메모리 GC로 진화)
```

### 3. UNWIND_STACK 내부 루틴

```typescript
private unwindStack(targetSP: number): void {
  // 현재 SP와 목표 SP의 차이 계산
  const gap = this.callStack.length - targetSP;

  // gap만큼 콜스택에서 프레임 제거
  for (let i = 0; i < gap; i++) {
    const frame = this.callStack.pop();
    // 각 프레임의 지역변수 정리
    frame.locals.forEach(local => {
      delete this.variables.get(local);
    });
  }

  this.log(`[UNWIND STACK] SP: ${this.callStack.length + gap} → ${this.callStack.length}`);
  this.log(`  → ${gap}개 프레임 파괴 완료`);
}
```

## 📝 테스트 케이스 상세

### TC_V8_4_001: SP Realignment (스택 높이 복구)

**목적**: THROW 전후의 스택 포인터(높이)가 정확히 일치하는가?

```freelang
let sp_before = __GET_STACK_DEPTH()

try {
  let a = 100        // Depth0 (깊이 0)

  fn depth1() {
    let b = 200      // Depth1 (깊이 1)

    fn depth2() {
      let c = 300    // Depth2 (깊이 2)
      throw new Exception("Unwind test")
      c = 999        // 실행 안 됨
    }
    depth2()
    b = 999          // 실행 안 됨
  }
  depth1()
  a = 999            // 실행 안 됨
}
catch (e) {
  // UNWIND 완료 후 여기 도착
}

let sp_after = __GET_STACK_DEPTH()

// 기대: sp_before == sp_after (스택 높이 완벽 복구)
if (sp_before == sp_after) {
  println(1)  // ✅
} else {
  println(-1) // ❌ 스택이 복구되지 않음
}
```

**기대 흐름**:
```
sp_before = 0 (TRY 진입 전)
↓
depth1() call: SP = 1
  depth2() call: SP = 2
    THROW 발생
    ↓ UNWIND_STACK(1) ← targetSP는 TRY 진입 시점
    ↓ SP: 2 → 1 → 0 (프레임 2개 파괴)
↓
CATCH 블록: SP = 0
↓
sp_after = 0
↓
sp_before == sp_after ✅
```

### TC_V8_4_002: FP Restoration (프레임 포인터 복구)

**목적**: 프레임 포인터가 정확히 이전 함수의 상태로 복구되는가?

```freelang
let fp_before = __GET_FP()

try {
  let local_var = 999

  fn nested() {
    let inner_var = 888
    throw new Exception("FP test")
  }
  nested()
}
catch (e) {
  let fp_catch = __GET_FP()

  // 기대: fp_catch == fp_before
  // (nested() 호출로 변경된 FP가 원상복구됨)
  if (fp_catch == fp_before) {
    println(1)  // ✅ FP 복구됨
  }
}
```

### TC_V8_4_003: Zombie Frame Prevention (좀비 프레임 방지)

**목적**: 파괴된 프레임의 변수에 접근하려는 시도가 차단되는가?

```freelang
try {
  fn create_local() {
    let zombie = 777
    throw new Exception("Zombie test")
  }
  create_local()

  // THROW 후 zombi 프레임 파괴됨
}
catch (e) {
  // zombie 변수는 메모리에서 제거됨
  // 접근 시도 시 "undefined variable" 에러 발생
  let x = zombie  // ❌ ReferenceError or undefined

  if (x == undefined || error_occurred) {
    println(1)  // ✅ 좀비 프레임 방지 성공
  }
}
```

## 🎯 설계 원칙

### "기록이 증명이다" in v8.4

```
v8.1: 핸들러 등록 ← "어디로 가는가?" 기록
v8.2: SP/FP/PC 저장 ← "어떤 상태로 가는가?" 증명
v8.3: PC로 점프 ← "지금 당장 간다!" 실행
v8.4: 스택 정리 ← "무엇을 버리는가?" 책임
         ↓
     "파괴의 기록"으로 나타남
```

### 메모리 일관성의 세 가지 약속
1. **SP 정합성**: 스택 높이가 정확히 복구됨
2. **FP 보존**: 프레임 포인터가 유효한 상태로 유지됨
3. **좀비 방지**: 파괴된 프레임의 변수에 접근 불가능
