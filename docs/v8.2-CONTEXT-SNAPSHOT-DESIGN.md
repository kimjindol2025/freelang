# FreeLang v8.2: Context Snapshot & Register Preservation

## 📌 개요

v8.1에서 "어디로 돌아갈 것인가"(핸들러 위치)를 정했다면, v8.2는 **"돌아갔을 때 어떤 상태인가"**(컨텍스트 스냅샷)를 완벽히 보존하는 단계입니다.

**핵심 원칙**: 기록이 증명이다 — 스냅샷은 "사고 전 정상 상태"의 유일한 무죄 증거

---

## 🏗️ 아키텍처 설계

### 1. ContextSnapshot 구조

```typescript
interface ContextSnapshot {
  savedSP: number;      // TRY 진입 시점의 Stack Pointer
  savedFP: number;      // TRY 진입 시점의 Frame Pointer
  savedPC: number;      // TRY 진입 시점의 Program Counter (복구 후 재시작점)
  timestamp: number;    // 스냅샷 생성 타임스탐프 (로깅용)
}
```

### 2. 확장된 HandlerFrame (v8.1 → v8.2)

```typescript
interface HandlerFrame {
  // ── v8.1: 기본 핸들러 정보 ──
  returnAddress: number;
  stackPointer: number;
  framePointer: number;
  catchBlockPC: number;
  tryStartPC?: number;
  exceptionVarName?: string;

  // ── v8.2: 컨텍스트 스냅샷 ──
  snapshot: ContextSnapshot;  // ← 새로 추가
}
```

### 3. 레지스터 백업 타이밍

```
TRY 블록 진입
  ↓
[SAVE_CONTEXT] 현재 레지스터 상태를 snapshot에 저장
  - SP (callStack.length)
  - FP (currentFramePointer)
  - PC (this.pc)
  - Timestamp
  ↓
try { ... 블록 실행 ... }
  ↓
정상 종료 또는 Exception 발생
  ↓
[RESTORE_CONTEXT] snapshot에서 읽어 레지스터 복구 (필요시)
  - callStack을 savedSP로 복원
  - currentFramePointer를 savedFP로 복원
  - (PC는 이미 핸들러에서 관리)
```

## 📝 테스트 케이스 상세

### TC_V8_2_001: SP Consistency

**목적**: TRY 진입 시 SP가 정확히 스냅샷에 저장되는가?

```freelang
// 초기 SP 기록
let initial_sp = __GET_STACK_DEPTH()

try {
  // 스택에 데이터 쌓음 (스택 오염)
  PUSH 100
  PUSH 200
  PUSH 300

  // 현재 SP (오염된 상태)
  let current_sp = __GET_STACK_DEPTH()

  // 스냅샷에 저장된 SP 조회
  let saved_sp = __GET_HANDLER_SP()

  // 검증: saved_sp == initial_sp (TRY 진입 시 상태)
  // 검증: saved_sp != current_sp (스택이 변했음을 증명)
}
catch (e) {
  // RESTORE_CONTEXT 후 SP가 initial_sp로 복구됨
}
```

**기대 흐름**:
```
initial_sp = 0
TRY 진입 → SAVE_CONTEXT (savedSP = 0)
PUSH 3번 → current_sp = 3
saved_sp = __GET_HANDLER_SP() → 0 (변화 없음!)
current_sp ≠ saved_sp ✅
```

### TC_V8_2_002: Frame Locking

**목적**: FP가 백업되어 함수의 지역변수 오염을 방지하는가?

```freelang
fn TestFunction() {
  let local_var = 999
  let initial_fp = __GET_FP()

  try {
    // FP 변경 시도 (다른 함수 호출 등)
    let saved_fp = __GET_HANDLER_FP()

    // 검증: saved_fp == initial_fp
    // (예외 발생 후 복구되어야 함)
  }
  catch (e) {
    // RESTORE_CONTEXT 후 FP가 복구됨
    let restored_fp = __GET_FP()

    // 검증: restored_fp == initial_fp
    // local_var는 여전히 999 (오염 없음)
  }
}
```

### TC_V8_2_003: Nested Safety

**목적**: 중첩 TRY 시 각 레벨의 스냅샷이 독립적인가?

```freelang
let outer_sp = __GET_STACK_DEPTH()

try {
  let saved_outer_sp = __GET_HANDLER_SP()

  try {
    let inner_sp = __GET_STACK_DEPTH()
    let saved_inner_sp = __GET_HANDLER_SP()

    // 검증:
    // saved_outer_sp == outer_sp (외부 스냅샷 보존)
    // saved_inner_sp == inner_sp (내부 스냅샷 독립)
    // saved_inner_sp ≠ saved_outer_sp (서로 다름)

    throw new Exception("inner error")
  }
  catch (e) {
    // 내부 스냅샷으로 복구
    // 외부는 계속 진행
  }

  // 외부 스냅샷이 여전히 유효
}
catch (e) {
  // 외부 스냅샷으로 복구
}
```

## 📊 스냅샷 저장 형식 (로깅)

```
[SAVE_CONTEXT] TRY 진입
  → SP (Stack Pointer): 0
  → FP (Frame Pointer): 0
  → PC (Program Counter): 5
  → Timestamp: 1771780900000
  → Snapshot 저장됨 (depth=1)

[TRY BLOCK EXECUTING] 블록 실행 중...

[EXCEPTION CAUGHT] 예외 감지!
  → 스냅샷 복구: SP=0, FP=0, PC=5

[RESTORE_CONTEXT] 레지스터 복구 완료
  → callStack 언와인드 (SP: 3 → 0)
  → FP 복구: 0
  → CATCH 블록 진입
```

## 🎯 설계 원칙

### "기록이 증명이다" (Snapshot as Evidence)

1. **스냅샷 생성**: TRY 진입 순간, 엔진의 완벽한 정상 상태 기록
2. **스냅샷 보존**: 핸들러 스택에 안전하게 보관
3. **스냅샷 복구**: exception 발생 시 이 상태로 강제 복구
4. **증거 가치**: "사고 전에 내 상태는 이랬다" — 엔진이 주장할 수 있는 유일한 증거

### v5.9 메모리 정렬과의 연결성

v5.9에서 구현한 **메모리 정렬과 포인터 연산**이 v8.2에서 진정한 가치를 발휘합니다:

- **SP 정합성**: 메모리 정렬의 정확성이 SP 값의 신뢰성 보장
- **FP 보존**: 지역변수의 정렬된 위치가 FP로 완벽히 복구됨
- **포인터 신뢰성**: 스냅샷의 SP/FP 값으로 모든 포인터가 유효한 상태로 복원됨
