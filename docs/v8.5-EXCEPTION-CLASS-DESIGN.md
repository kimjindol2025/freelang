# FreeLang v8.5: Exception Class & Objectification

## 📌 개요

v8.4까지 스택을 정리했다면, v8.5는 예외를 **"단순한 신호"에서 "데이터를 가진 존재"로 격상**하는 단계입니다.

**핵심 원칙**: "이름표가 붙은 에러"
- Before: "문제가 생겼다" (문자열)
- After: "누가, 왜, 어디서" 문제를 일으켰는가 (객체)

---

## 🏗️ 아키텍처 설계

### 1. Built-in Exception 클래스

```typescript
class Exception {
  // 내부 심볼로 엔진이 관리
  __type: 'object'
  __class: 'Exception'
  __typeName: 'Exception'

  // 필드 (스택에 저장)
  Message: string;        // 에러 메시지
  Code: int;              // 에러 코드 (HTTP 같은 숫자 구분)
  Timestamp: string;      // 발생 시각
  Location: string;       // 발생 위치 (파일명:라인)

  // 메서드
  GetTrace(): string;     // 스택 트레이스 반환
  ToString(): string;     // 문자열 표현
}
```

### 2. Exception Registry

```
엔진 초기화 시:
1. Exception이라는 구조체 정의 (필드: Message, Code, Timestamp, Location)
2. classTable에 Exception 등록
3. 기본 인스턴스 생성 루틴 구성
```

### 3. THROW 연산 업그레이드

**Before (v8.3):**
```
THROW new Exception("Error message")
→ JavaScript Error 객체 생성
→ 외부 try-catch로 전파
```

**After (v8.5):**
```
THROW exceptionObject
→ exceptionObject의 모든 필드 보존
→ handlerStack에 기록된 PC로 점프
→ CATCH 블록에서 객체 접근 가능
```

## 📝 테스트 케이스 상세

### TC_V8_5_001: Object Survival (객체 생존)

**목적**: 언와인딩 도중에도 예외 객체가 메모리에 생존하는가?

```freelang
try {
  let err = new Exception()
  err.Message = "Test Error"
  err.Code = 500

  throw err

  // 실행 안 됨
  err.Code = 999
}
catch (e) {
  // 기대: e.Code == 500 (원래 값 보존)
  // 불가능한 것: e.Code == 999 (할당 안 됨)
  if (e.Code == 500) {
    println(1)  // ✅ 객체 생존
  } else {
    println(0)  // ❌ 객체가 손상됨
  }
}
```

**기대 흐름**:
```
err 객체 생성 @ 0x5000
  Message = "Test Error"
  Code = 500

THROW err
  ↓ UNWIND_STACK (중간 프레임 제거)
  → err 객체는 힙에 그대로 살아있음 (참조 유지)
  ↓ PC_REDIRECT

CATCH 블록에서 e = err
  ↓ e.Code 접근
  → 500 (원래 값)
```

### TC_V8_5_002: Field Accessibility (필드 접근성)

**목적**: CATCH 블록에서 예외 객체의 멤버 변수에 접근 가능한가?

```freelang
try {
  let ex = new Exception()
  ex.Message = "Access Denied"
  ex.Code = 403

  throw ex
}
catch (e) {
  // 예외 객체의 필드들에 접근
  let msg = e.Message   // "Access Denied"
  let cod = e.Code      // 403

  if (msg == "Access Denied" && cod == 403) {
    println(1)  // ✅ 필드 접근 성공
  } else {
    println(0)
  }
}
```

### TC_V8_5_003: Default Constructor (기본 생성자)

**목적**: Exception 클래스가 인자 없이 생성 가능한가?

```freelang
try {
  // new Exception() 호출 — 기본 생성자
  let exception = new Exception()

  // 필드 설정 (선택사항)
  exception.Message = "Generic Error"
  exception.Code = 0

  throw exception
}
catch (e) {
  // 기대: e가 유효한 Exception 객체
  if (e.__class == "Exception") {
    println(1)  // ✅ 인스턴스화 성공
  } else {
    println(0)
  }
}
```

## 🎯 설계 원칙

### "기록이 증명이다" in v8.5

```
v8.1: 핸들러 등록 ← 위치 기록
v8.2: SP/FP/PC 저장 ← 상태 기록
v8.3: PC로 점프 ← 행동 기록
v8.4: 스택 정리 ← 파괴 기록
v8.5: 객체 속성 ← 내용 기록
         ↓
      "이름표가 붙은 에러"
```

### 예외 객체의 의미

```
단순 신호 ("Error") vs 데이터를 가진 존재 (Exception 객체)

예외 객체는:
1. 메모리에 존재 (객체)
2. 속성을 가짐 (Message, Code)
3. 메서드를 가짐 (GetTrace)
4. 식별 가능 (객체 ID)
5. 추적 가능 (스택 트레이스)

→ 나중에 v10에서 정교한 로깅 시스템의 기초
```

## 💡 핵심 통찰

> 예외는 단순한 에러 메시지가 아닙니다.
> 그것은 **"발생한 일의 전체 기록"**입니다.
>
> Exception 객체가 되면, 에러는 이름표가 붙은 존재가 되고,
> 그 객체를 추적함으로써 문제의 원인을 찾을 수 있습니다.
>
> 이것이 v8.5가 하는 일입니다.
