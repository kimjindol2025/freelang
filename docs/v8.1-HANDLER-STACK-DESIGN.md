# FreeLang v8.1: Handler Stack 설계 문서

**목표**: 예외 발생 시 돌아갈 피난처를 기록하는 독립적인 스택 구축

---

## 1️⃣ 핸들러 프레임 (Handler Frame) 구조

### 데이터 정의
```typescript
interface HandlerFrame {
  returnAddress: number;    // PC: CATCH 블록 시작 주소
  stackPointer: number;     // SP: TRY 진입 시점의 데이터 스택 깊이
  framePointer: number;     // FP: 현재 함수의 지역 변수 시작 지점
  catchBlockPC: number;     // CATCH 블록의 bytecode PC
  exceptionVarName?: string; // 예외 변수명 (선택사항)
}
```

### 메모리 위치
- **일반 데이터 스택**: 함수 인자, 지역 변수 → callStack[]
- **핸들러 스택**: TRY/CATCH 피난처 → handlerStack[]
- **두 스택은 완전히 독립적임** (충돌 방지)

---

## 2️⃣ 엔진 수정 사항

### A. 새로운 바이트코드

#### PUSH_HANDLER [targetAddr]
- **역할**: TRY 블록 진입 시 핸들러 프레임을 스택에 추가
- **실행 흐름**:
  1. 현재 SP 값 기록
  2. 현재 FP 값 기록
  3. targetAddr (CATCH 블록 PC) 저장
  4. handlerStack에 새 프레임 푸시

**로깅**:
```
[PUSH_HANDLER] TRY 진입 (PC=100)
  → HandlerFrame {
      returnAddress: 250,      // CATCH 블록 시작 주소
      stackPointer: 3,         // 현재 콜스택 깊이
      framePointer: 128,       // FP 저장
      catchBlockPC: 250
    }
  → handlerStack.length = 1 ✅
```

#### POP_HANDLER
- **역할**: TRY 블록이 정상 종료되었을 때, 등록된 핸들러 제거
- **실행 흐름**:
  1. 현재 SP가 저장된 SP와 일치하는지 검증
  2. FP가 저장된 FP와 일치하는지 검증
  3. handlerStack 맨 위 프레임 제거

**로깅**:
```
[POP_HANDLER] TRY 정상 종료
  → 저장 SP: 3, 현재 SP: 3 ✅
  → handlerStack.length = 0 ✅
```

### B. PCInterpreter 수정 항목

1. **필드 추가**:
   ```typescript
   private handlerStack: HandlerFrame[] = [];
   private MAX_HANDLER_DEPTH = 100;
   ```

2. **__GET_HANDLER_COUNT() 내장 함수**:
   ```typescript
   this.variables.set('__GET_HANDLER_COUNT', () => {
     return this.handlerStack.length;
   });
   ```

3. **PUSH_HANDLER 구현** (TRY 문 시작):
   ```typescript
   case 'TryStatement': {
     const handlerFrame: HandlerFrame = {
       returnAddress: /* CATCH 블록 PC 계산 */,
       stackPointer: this.callStack.length,
       framePointer: this.currentFramePointer,
       catchBlockPC: /* CATCH 블록 시작 */
     };
     this.handlerStack.push(handlerFrame);
     this.log(`[PUSH_HANDLER] TRY 진입 (depth=${this.handlerStack.length})`);

     // TRY 블록 실행
     try {
       for (const stmt of node.tryBlock) {
         this.eval(stmt);
       }
     } catch (e) {
       // v8.2 에서 처리
     }

     // 정상 종료 시 POP
     this.handlerStack.pop();
     this.log(`[POP_HANDLER] TRY 정상 종료 (depth=${this.handlerStack.length})`);
   }
   ```

4. **Return 문 수정** (함수 종료 시 자동 POP):
   ```typescript
   case 'ReturnStatement': {
     // 함수 종료 시 핸들러도 함께 정리
     // (동일 깊이 핸들러만 pop)
   }
   ```

## 3️⃣ 테스트 케이스: TC_V8_1_HANDLER_LIFECYCLE

### 테스트 목표
- Handler Stack의 PUSH/POP이 정확하게 작동하는지 검증
- 중첩 TRY의 깊이 추적이 정확한지 확인
- 정상 종료 시 모든 핸들러가 제거되는지 확인

### FreeLang 코드
```freelang
// [V8.1-HANDLER-LIFECYCLE-TEST]

// 1. 초기 상태
let start_count = __GET_HANDLER_COUNT()
println(start_count)  // 0

// 2. 단일 TRY 진입
try {
  let mid_count = __GET_HANDLER_COUNT()
  println(mid_count)  // 1
}

// 3. 중첩 TRY
try {
  println(__GET_HANDLER_COUNT())  // 1

  try {
    println(__GET_HANDLER_COUNT())  // 2
  }

  println(__GET_HANDLER_COUNT())  // 1
}

// 4. 최종 상태
let final_count = __GET_HANDLER_COUNT()
println(final_count)  // 0

println(999)  // 모든 테스트 통과
```

### 예상 출력
```
0  # 초기 상태
1  # 단일 TRY 진입
1  # 중첩 TRY 외부
2  # 중첩 TRY 내부
1  # 중첩 TRY 외부 복귀
0  # 최종 상태
999  # 성공
```

## 4️⃣ 무결성 체크리스트

| 검증 항목 | 상세 내용 | 기대 결과 |
|----------|---------|---------|
| **Overflow Check** | 핸들러 스택이 MAX_HANDLER_DEPTH(100)을 넘을 때 | `[HANDLER ERROR] 핸들러 스택 오버플로우` 예외 |
| **Clean Exit** | TRY 내부에서 RETURN 시 핸들러 POP | `[POP_HANDLER]` 로그 + 스택 정리 |
| **Metadata Integrity** | 저장된 SP/FP와 현재 값 일치 확인 | `[HANDLER INTEGRITY OK]` 로그 |
| **Nesting Depth** | 최대 10단계 중첩 TRY 지원 | 모두 정상 작동 |
| **Early Exit** | BREAK/CONTINUE로 TRY 탈출 | 핸들러 자동 POP |

## 7️⃣ 핵심 원칙

### "기록이 증명이다"
- **모든 PUSH/POP은 로그로 기록됨**
- 스택의 현재 깊이 추적 → `__GET_HANDLER_COUNT()`로 검증 가능
- 로그가 없으면 일어나지 않은 것

### 정상 종료 시 반드시 POP
⚠️ **매우 중요**:
- POP하지 않으면 다음 예외가 발생할 때 **잘못된 위치로 점프**
- 이를 방지하기 위해:
  1. TRY 블록 정상 완료 → 즉시 POP
  2. RETURN으로 함수 종료 → 핸들러들을 깊이 기반으로 정리
  3. BREAK/CONTINUE → 해당 깊이까지 핸들러 정리

v8.1 Handler Stack은:
- ✅ **독립적인 스택**: 데이터 스택과 완전히 분리

- ✅ **중첩 지원**: 임의 깊이의 중첩 TRY 가능
- ✅ **메타데이터 추적**: SP/FP로 스택 무결성 보증
